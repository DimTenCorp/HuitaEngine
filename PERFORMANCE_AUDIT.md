# Производственный аудит производительности HuitaEngine

Дата: 2026-04-15

## Критические проблемы (наибольшее влияние на FPS/frametime)

1. **Полная пересборка динамического коллайдера дверей каждый кадр**  
   В `Engine::updateDoors` на каждом кадре заново строится `std::vector<Triangle> doorTriangles`, для каждой двери заново трансформируются все вершины, пересчитываются нормали и затем вызывается `meshCollider->updateDynamicTriangles(doorTriangles)`.  
   **Риск:** выраженный CPU bottleneck при большом числе дверей и высоком количестве треугольников.

2. **Создание/удаление OpenGL буферов в каждом кадре для рендера дверей**  
   В `LightmappedRenderer::renderDoors` каждый вызов создаёт `tempVAO/tempVBO/tempEBO` через `glGen*`, потом удаляет их через `glDelete*`.  
   **Риск:** дорогой драйверный overhead и микрофризы (allocator churn на стороне GPU/driver).

3. **Сортировка прозрачных draw call'ов каждый кадр через выделения памяти**  
   И в `Renderer::renderTransparentFacesForward`, и в `LightmappedRenderer::renderWorld` в каждом кадре формируется новый `std::vector<SortedDrawCall>`, заполняется и сортируется `std::sort`.  
   **Риск:** лишняя работа CPU + частые аллокации при большом числе прозрачных полигонов.

## Высокий приоритет

4. **Отсутствие кэширования матрицы projection**  
   `glm::perspective(...)` считается в нескольких путях рендера практически каждый кадр (`Engine::render`, `Renderer::renderWorld`, `LightmappedRenderer::renderWorld`).  
   **Риск:** небольшой, но постоянный CPU overhead; на слабых CPU и при росте количества pass'ов влияние становится заметнее.

5. **Высокая цена draw-call driven пайплайна мира**  
   Рендер мира выполняется поштучно по `faceDrawCalls`/`drawCalls` с `glDrawElements` на каждый участок. Даже при оптимизации смены текстур количество draw calls может быть высоким.  
   **Риск:** CPU-bound рендер на сценах с большим количеством поверхностей.

6. **Постоянные state changes OpenGL между pass'ами без минимизации**  
   В рендере регулярно повторяются переключения `glEnable/glDisable`, `glDepthFunc`, `glDepthMask`, blend-состояний. Часть необходима, но часть потенциально избыточна и может быть сокращена через state cache.  
   **Риск:** driver overhead, особенно на старых драйверах/встроенной графике.

## Средний приоритет

7. **Пересчёт расстояния прозрачных объектов по одной вершине draw call'а**  
   Для сортировки прозрачности берётся первая вершина. Это дешево, но даёт неверный порядок для крупных полигонов и может вызывать overdraw/артефакты -> повторные попытки исправления через усложнение пайплайна.  
   **Риск:** компромисс между качеством и производительностью; при артефактах обычно переходят к более дорогим решениям.

8. **Синхронная загрузка тяжёлых ресурсов в runtime потоке**  
   Загрузка BSP/WAD/сборка геометрии и вспомогательных структур выполняется синхронно в основном потоке загрузки карты.  
   **Риск:** большие spike'и времени кадра при смене карт, отсутствие асинхронности.

## Рекомендуемый план оптимизации

### Этап 1 (быстрые победы, 1-2 дня)
- Перевести `renderDoors` на постоянные (persistent) VAO/VBO/EBO, создаваемые один раз при init.
- Для прозрачных объектов хранить `sorted` буфер как поле класса и переиспользовать память (`clear` + `reserve` один раз).
- Кэшировать projection matrix и пересчитывать только при изменении FOV/разрешения.
- Добавить простой OpenGL state cache (не вызывать `gl*` если состояние не меняется).

### Этап 2 (сильный эффект, 3-5 дней)
- Двери: обновлять динамический коллайдер только при фактическом изменении transform двери (dirty flag), а не каждый кадр.
- Двери: предвычислить локальные треугольники/нормали один раз; в апдейте применять только transform.
- Для draw calls мира: батчинг по материалу/текстуре и/или MDI (если целевой GL позволяет).

### Этап 3 (архитектурный, 1-2 недели)
- Ввести фрустум-каллинг по батчам/leaf'ам BSP на CPU до этапа draw.
- Рассмотреть GPU-driven подход (indirect + compute culling) для больших сцен.
- Профилировать с таймерами GPU/CPU (RenderDoc + Tracy/Remotery) и закрепить performance budget.

## Что измерять после правок

- CPU frametime (p50/p95/p99) отдельно для:
  - updateDoors
  - transparent sort
  - world submit (draw-call loop)
- Количество draw calls/triangles в кадре.
- Время GPU на opaque/transparent/doors pass.
- Частоту аллокаций в кадре (должна резко снизиться после переиспользования буферов).

## Ожидаемый эффект

При типичной HL/Quake-сцене с большим числом дверей и прозрачных поверхностей:
- устранение per-frame glGen/glDelete и лишних аллокаций: **минус 0.3-2.0 ms CPU/frame**;
- dirty-update для коллайдера дверей: **минус 0.5-4.0 ms CPU/frame** (зависит от геометрии);
- частичный батчинг и state-cache: **минус 0.5-3.0 ms CPU/frame**.

Итого реалистичный потенциал: **~1.3-9.0 ms CPU/frame** на проблемных картах.
